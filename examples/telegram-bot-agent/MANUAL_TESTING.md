## 1. Подготовка окружения

### Необходимые ресурсы

- **Telegram-аккаунт владельца** (основной тестировщик)
- **Telegram-аккаунт обычного пользователя** (второй телефон/аккаунт)
- **Тестовый Telegram-канал** (бот должен быть админом)
- **Тестовая Telegram-группа** (бот должен быть админом)
- **Рабочий агент-воркер** (например, `simple-prompt-agent`)

### Запуск бота

**Вариант A — Polling (локальная разработка, без вебхуков):**

```bash
cd examples/telegram-bot-agent
cp .dev.vars.example .dev.vars
# Заполнить TELEGRAM_BOT_TOKEN и AGENT_URL в .dev.vars
pnpm dev:polling
```

> Внимание: polling использует MockKV (данные хранятся в памяти и пропадают при перезапуске).

**Вариант B — Wrangler dev (локально с KV):**

```bash
cd examples/telegram-bot-agent
pnpm dev
# Затем настроить вебхук:
# curl "https://api.telegram.org/bot<TOKEN>/setWebhook?url=<NGROK_URL>/webhook"
```

**Вариант C — Deploy на Cloudflare:**

```bash
cd examples/telegram-bot-agent
pnpm deploy
# Настроить вебхук:
# curl "https://api.telegram.org/bot<TOKEN>/setWebhook?url=https://telegram-bot-agent.<subdomain>.workers.dev/webhook"
```

### Проверка работоспособности

```bash
# Health check
curl https://<BOT_URL>/health

# Проверка вебхука
curl "https://api.telegram.org/bot<TOKEN>/getWebhookInfo"
```

Ожидаемый результат health: `{"status":"ok"}` или аналог.

## 9. Тест-кейсы: Модерация

### Подготовка
- Добавить бота в тестовую группу как **администратора** с правами удаления сообщений, бана, мьюта.

### TC-9.1: Включение модерации

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|
| 1 | В группе отправить `/moderate` (владелец или админ) | Настройки модерации с кнопками |
| 2 | Нажать «🟢 Enable» | Toast: «Moderation enabled ✅», кнопка меняется на «🔴 Disable» |

### TC-9.2: Настройка детекции

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|
| 1 | В настройках модерации переключить Spam, Scam, Hate, Flood, Links | Каждый переключатель меняет состояние |

### TC-9.3: Настройка действий

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|
| 1 | Нажать «⚙️ Configure Actions» | Меню действий по категориям |
| 2 | Для Spam выбрать «🗑️ Delete» | Toast: «Action set to delete» |
| 3 | Для Scam выбрать «🚫 Ban» | Toast: «Action set to ban» |

### TC-9.4: Тест спам-детекции

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|
| 1 | С аккаунта НЕ-владельца отправить спам-сообщение (типа «BUY NOW! CLICK bit.ly/xxx FREE MONEY!!!») | Сообщение удалено/предупреждение (в зависимости от настроек) |

### TC-9.5: Тест flood-детекции

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|
| 1 | С аккаунта НЕ-владельца быстро отправить 5+ сообщений подряд | Flood обнаружен, действие применено |

### TC-9.6: Тест whitelist

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|
| 1 | В группе ответить на сообщение пользователя: `/whitelist add` | `✅ Added <username> to whitelist.` |
| 2 | Этот пользователь отправляет спам | Сообщение **не** модерируется |
| 3 | Отправить `/whitelist list` | Пользователь в списке |
| 4 | Отправить `/whitelist remove <userId>` | `✅ Removed user ID <id> from whitelist.` |

### TC-9.7: Статистика модерации

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|
| 1 | Отправить `/modstats` | Выбор группы (если несколько) или статистика |
| 2 | Проверить: кол-во по категориям, по действиям, за 24ч | Данные отображаются |

### TC-9.8: Выключение модерации

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|
| 1 | В настройках нажать «🔴 Disable» | Toast: «Moderation disabled ❌» |
| 2 | Отправить спам-сообщение | Сообщение **не** удаляется |

### TC-9.9: Эскалация предупреждений

| # | Шаг | Ожидаемый результат |
|---|------|---------------------|

# Часть II — Веб-дашборд (Playground)

## 18. Подготовка дашборда

### Архитектура

Дашборд — встроенный веб-интерфейс в single-worker (`examples/single-worker`). Он содержит:
- **Playground (Chat)** — чат с AI-агентами через веб-интерфейс
- **Manager Dashboard** — панель управления Telegram-ботом (появляется только при наличии `TELEGRAM_BOT_TOKEN`)

### Запуск single-worker

```bash
cd examples/single-worker
cp .dev.vars.example .dev.vars
# Заполнить: TELEGRAM_BOT_TOKEN, OPENAI_API_KEY (или другой AI provider), AGENT_URL
pnpm dev
```

Дашборд доступен по адресу: `http://localhost:8787/` (или deployed URL).

### Получение PIN для дашборда

PIN для доступа к Manager Dashboard получается через Telegram-бота:
1. Отправить `/start` или `/pin` боту в DM
2. Бот выдаст числовой PIN
3. Этот PIN используется для авторизации в веб-дашборде

### Необходимые ресурсы

- Браузер (Chrome/Firefox/Safari)
- Рабочий single-worker с настроенным `TELEGRAM_BOT_TOKEN`
- PIN от Telegram-бота
- Тестовый канал и группа (для полного тестирования Chat Settings)

## Примечания

- **Порядок тестирования:** рекомендуется начать с разделов 2-4 (базовая функциональность Telegram), затем 18-19 (запуск и авторизация дашборда), затем 5-8 (контент-менеджмент), 20-24 (дашборд Overview/Config/Settings), затем 9-10 (модерация/проактив), 25-27 (чат в Playground), затем 11-13 (kanban/cron), 28-29 (API и edge cases), и наконец 30 (кросс-проверка дашборд ↔ Telegram).
- **KV данные:** при использовании polling (MockKV) данные теряются при перезапуске. Для полноценного тестирования используйте `wrangler dev` или deploy.
- **Два аккаунта:** для тестов модерации и разграничения прав необходимо два Telegram-аккаунта.
- **Тестовый канал:** бот должен быть администратором канала с правом публикации.
- **Тестовая группа:** бот должен быть администратором с правами удаления сообщений, бана и мьюта.
- **Браузеры:** для тестирования дашборда рекомендуется проверить в Chrome, Firefox и Safari (или хотя бы в Chrome).
- **localStorage:** Playground хранит историю чатов и настройки агентов в localStorage. Очистка localStorage сбрасывает эти данные.
- **PIN:** PIN для Manager Dashboard выдаётся через Telegram-бота. Без PIN доступен только Chat tab.
